Technical Design Document: Android Jetpack Compose E-Reader Architecture Featuring Readium, Synchronized Text-to-Speech, and Nextcloud WebDAV Integration
The development of a highly accessible, cross-device synchronized e-reader application for the Android platform requires a sophisticated architectural approach. Modern Android development paradigms heavily favor declarative user interfaces and reactive data streams, which must be carefully reconciled with the complex, often imperative nature of parsing and rendering Electronic Publications (EPUBs). This technical design document provides an exhaustive, ground-up blueprint for constructing such an application. The system relies on the Readium Kotlin Toolkit for document parsing and rendering, Jetpack Compose for the user interface, the native Android Text-to-Speech (TTS) engine coupled with Readium’s Decorator API for synchronized highlighting, Android Room for local persistence, and the dav4jvm library operating within Android's WorkManager for seamless Nextcloud WebDAV synchronization. Furthermore, this document outlines the foundational architecture required to achieve a future stretch goal of utilizing server-side generated Synchronized Multimedia Integration Language (SMIL) for dynamic Media Overlays.
Designed specifically as a comprehensive architectural analysis and a sequential guide for Large Language Model (LLM) assisted development, the implementation phases detailed herein are structured to be directly translated into modular prompts. This ensures the LLM maintains context without hallucinating dependencies or diverging from the established architectural patterns.
Core Architectural Paradigm and Dependency Management
The application architecture is strictly grounded in the Model-View-ViewModel (MVVM) pattern, heavily leveraging Kotlin Coroutines and StateFlow to maintain a reactive, unidirectional data pipeline. This pipeline connects the local Room database, the Readium rendering engine, and the Jetpack Compose user interface. The separation of concerns is enforced to ensure that the computationally expensive logic of parsing EPUB packages, tokenizing text for speech synthesis, and managing remote WebDAV synchronization does not block the main UI thread.
Readium Kotlin Toolkit 3.x Migration and Setup
The primary dependency for parsing and rendering EPUBs is the Readium Kotlin Toolkit. The ecosystem has recently transitioned to version 3.x (with active versions such as 3.1.2), which introduces significant architectural shifts that must be accounted for during initialization.1 Previously, Readium utilized a local HTTP server to serve EPUB assets to its internal web views. In version 3.0 and above, this local server has been removed in favor of direct asset fetching via custom web view interceptors, simplifying the network stack but requiring careful attention to base URL configurations.2 Furthermore, the toolkit has migrated entirely to a single monorepo (kotlin-toolkit) and is now distributed exclusively via Maven Central, deprecating previous JitPack distributions.3
To integrate the toolkit, the application's build.gradle.kts must target Android SDK 34 or higher, utilize Gradle 8.6 or higher, and compile with Kotlin 1.9.24 or a more recent version.1 The group ID for dependencies is org.readium.kotlin-toolkit, and the project requires specific modules to function.4

Application Manifest and Permissions
The integration of background audio, text-to-speech, and remote network synchronization necessitates explicit permission declarations within the AndroidManifest.xml. To communicate with the Nextcloud server, the standard android.permission.INTERNET is mandatory. Because synchronization tasks and prolonged TTS playback must survive application backgrounding, android.permission.WAKE_LOCK is required.7
Furthermore, to ensure the application can successfully query and bind to the native Android TTS engine on devices running Android 11 (API level 30) and above, the manifest must declare an intent query for android.intent.action.TTS_SERVICE.8 The absence of this query element will result in the package manager obscuring installed TTS engines from the application, causing fatal initialization errors. Finally, due to the target SDK being 34 and the utilization of Readium's Jetpack Media3 integration for background audio playback, the android.permission.FOREGROUND_SERVICE_MEDIA_PLAYBACK must be explicitly declared to satisfy modern Android foreground service restrictions.2
Declarative UI Construction: Jetpack Compose and Fragment Interoperability
The primary challenge in constructing the reading interface lies in reconciling the declarative nature of Jetpack Compose with the imperative, lifecycle-driven architecture of Readium's EpubNavigatorFragment. While the surrounding application architecture—such as library views, settings menus, and statistical dashboards—is written entirely in Compose, the actual rendering of the EPUB content requires embedding an Android Fragment.9
The Interoperability Bridge
Bridging these two distinct UI paradigms requires the utilization of AndroidViewBinding, provided by the androidx.compose.ui:ui-viewbinding library.10 The application defines an XML layout containing a standard FragmentContainerView. Within the Jetpack Compose hierarchy, the AndroidViewBinding composable inflates this layout and provides access to the underlying FragmentManager.9
The Readium Streamer parses the EPUB file to generate a Publication object, which is then passed to the EpubNavigatorFragment.createFactory() method.5 State management across this bridge is highly critical. Compose thrives on unidirectional data flow and frequent recompositions. If the fragment instantiation logic is not carefully isolated, every recomposition of the reading screen will attempt to recreate the EpubNavigatorFragment, leading to severe memory leaks, UI flickering, and complete loss of the user's reading position.10
To mitigate this, the application must utilize Compose remember blocks to cache the Publication, the factory instance, and the fragment itself. The AndroidViewBinding is wrapped in an effect block that only executes when the underlying Publication explicitly changes, ensuring the fragment remains stable throughout the lifecycle of the reading activity.10
Dynamic Preferences via the Preferences API
Readium 3.x introduces a completely overhauled Preferences API that perfectly complements the Compose state-hoisting architecture. Navigators are now configured dynamically via the Configurable interface. Instead of imperatively mutating individual settings (which is hazardous in a reactive environment), the application submits an immutable Preferences object to the navigator.13
In a Compose architecture, user preferences—such as font size, theme, and font family—are managed within a ViewModel and exposed as a StateFlow<EpubPreferences>. The toolkit provides a PreferencesEditor to assist in generating valid preference objects based on hardware and publication constraints.13 When a user interacts with a Compose settings menu to increase the font size, the ViewModel emits a new EpubPreferences state. The Compose UI observes this state change and triggers a LaunchedEffect, executing navigator.submitPreferences(preferences). The EpubNavigatorFragment then dynamically recalculate its settings and updates the internal web view presentation without requiring a fragment reload.13
A critical nuance of this API is the distinction between value and effectiveValue. The value represents the user's requested preference, which may be null or unsupported. The effectiveValue represents the actual value adopted by the navigator after evaluating constraints. The Compose UI must bind its visual state (e.g., slider positions or toggle switches) using the pattern preference.value?: preference.effectiveValue to accurately reflect the internal state of the rendering engine.13
Text-to-Speech Orchestration and Synchronized Highlighting
The integration of Text-to-Speech functionality paired with real-time visual highlighting elevates the application from a simple document viewer to a highly accessible reading platform. Readium 3.0 decoupled its audio features, promoting the PublicationSpeechSynthesizer into a fully standalone TtsNavigator that implements the MediaNavigator interface.14 This separation allows the TTS engine to operate in the background indefinitely, independent of the visual fragment.8
Tokenization and Synthesis
The TTS initialization phase requires creating an AndroidTtsNavigatorFactory, providing the application context and the parsed Publication.15 The TtsNavigator relies on a tokenizer to parse the HTML textual content of the EPUB into manageable audio chunks, termed "utterances".8 By default, Readium utilizes a sentence-based tokenizer, but the architecture allows the injection of custom tokenizers. For instance, if the application requires word-by-word highlighting, a DefaultTextContentTokenizer configured with TextUnit.Word can be supplied to the factory.15
Because the TtsNavigator defaults to the native Android TTS engine, the application must robustly handle missing language data. If a user attempts to read a French EPUB but lacks the French voice data pack, the TtsNavigator emits a Failure.EngineError. The application's ViewModel observes the navigator.playback stream; upon detecting an AndroidTtsEngine.Error.LanguageMissingData error, it calls AndroidTtsEngine.requestInstallVoice(context) to trigger the system intent, prompting the user to download the necessary files.15
Visual Synchronization and the Decorator API
To fulfill the requirement of reading aloud while highlighting the text, the application must deeply synchronize the background TtsNavigator with the foreground EpubNavigatorFragment. When the user initiates playback, the application utilizes the VisualNavigator.firstVisibleElementLocator() API to determine the exact structural element currently rendered on the screen. This locator is passed as the initial seed to the TtsNavigator, ensuring the audio begins precisely where the user is looking.15
As the synthetic voice reads the text, the TtsNavigator emits a continuous StateFlow of its current position, accessible via the ttsNavigator.location property.15 The application must map this spatial progression to visual highlights utilizing Readium’s Decorator API. The EpubNavigatorFragment implements the DecorableNavigator interface, granting the application the ability to draw arbitrary shapes, underlines, or background colors over specific locators within the publication.16
The implementation requires a Coroutine collecting the ttsNavigator.location stream. A distinctUntilChanged() filter is applied to prevent redundant DOM manipulations. The locator of the current utterance is extracted, and a Decoration object is constructed.15 A decoration consists of a unique identifier (e.g., "tts-utterance"), the target locator, and a Decoration.Style.Highlight style with a specific color tint.15
The application invokes navigator.applyDecorations(listOf(decoration), group = "tts").15 Utilizing a dedicated decoration group named "tts" is an essential architectural decision. Grouping decorations ensures that as the TTS engine moves to the next sentence, the previous highlight is automatically cleared and replaced without interfering with user-generated annotations or active search result highlights.16 Furthermore, if the utteranceLocator progresses outside the current viewport bounds of the EpubNavigatorFragment, the ViewModel must programmatically trigger a goForward() or go(locator) command on the visual navigator to automatically turn the page, ensuring a continuous reading experience.11
Local Persistence Strategy: Android Room Architecture
To fulfill the requirements of persisting reading statistics, backing up exact reading positions, and caching publication metadata, a highly reliable local database is required. Android Room, which provides a strongly typed abstraction layer over SQLite, serves as the local source of truth. Room's native integration with Kotlin Coroutines and Flow ensures that database changes are automatically pushed to the Jetpack Compose UI without manual querying.18
The database schema is structured around three primary entities, enforcing relational integrity through foreign keys.

The Readium toolkit heavily utilizes custom data types, such as the Locator object, which represents a highly specific pointer within an EPUB (containing HREFs, progression percentages, and DOM text locations). Room cannot natively store these complex objects. Therefore, the application must implement Room TypeConverter classes to serialize the Locator object into a JSON string during database insertion, and deserialize it back into a valid Kotlin object during retrieval.19 Furthermore, Readium 3.x has removed legacy JVM dependencies such as java.util.Date in favor of a custom Instant type, requiring an additional TypeConverter for timestamp management.20
Statistical tracking is orchestrated by a session management service within the repository layer. When a user opens an EPUB, a new ReadingSessionEntity is instantiated with a start timestamp. As the user reads, the EpubNavigatorFragment fires onProgressionChanged() events.17 The application observes these events to continuously update the ReadingPositionEntity and increment the pagesTurned counter in memory. When the user closes the publication or the application is sent to the background, the session service calculates the durationSeconds, commits the finalized ReadingSessionEntity to the Room database, and updates the totalProgression on the BookEntity.17 The Compose UI, observing a Flow<List<ReadingSessionEntity>>, automatically recomposes to render updated statistical charts.
Remote Synchronization: Nextcloud WebDAV Integration
The requirement to securely backup reading positions, statistics, and application data to a Nextcloud instance introduces the complexity of remote file synchronization. Nextcloud relies fundamentally on the Web Distributed Authoring and Versioning (WebDAV) protocol, an HTTP extension that facilitates the creation, reading, and remote editing of files.21
While Nextcloud provides its own Android Library and Single Sign-On (SSO) applications, these tools are heavily coupled with the Nextcloud Files app and introduce significant overhead for a lightweight background synchronization task.23 Instead, the architecture utilizes dav4jvm, a dedicated WebDAV client developed by BitfireAT (and utilized by Nextcloud itself), to execute asynchronous HTTP requests via OkHttp directly within Kotlin Coroutines.24
Authentication and Security
Authentication with Nextcloud over WebDAV requires careful handling of user credentials. Modern Nextcloud instances frequently employ Two-Factor Authentication (2FA) or integrate with SSO providers such as Active Directory or SAML 2.0 (e.g., CyberArk, Shibboleth).26 Utilizing a user's primary credentials for a background synchronization script under these security models is prone to failure and represents an anti-pattern.
The application resolves this by mandating the use of Nextcloud App Passwords. The user generates a dedicated, revocable application password within their Nextcloud security settings.21 The Android application securely stores the Nextcloud server URL, the username, and the App Password using EncryptedSharedPreferences. The dav4jvm library is then initialized with an OkHttp client configured to use a BasicDigestAuthHandler, supplying these stored credentials to authenticate against the WebDAV endpoint securely.25
Payload Generation and WorkManager Execution
The synchronization strategy relies on exporting the Room database contents into structured JSON payloads. When a synchronization event is triggered, the application queries the local ReadingPositionEntity and ReadingSessionEntity tables, aggregates the data, and serializes it into a JSON string. This string is converted into an OkHttp RequestBody using toRequestBody("application/json".toMediaType()).25
The application constructs a DavCollection pointing to a dedicated directory on the Nextcloud server, adhering to the required WebDAV URL structure: https://example.com/nextcloud/remote.php/dav/files/USERNAME/EreaderSync/.21 The davCollection.put() suspending function is invoked to upload the payload. Because WebDAV operates over standard HTTP verbs, the PUT request seamlessly creates or overwrites the backup file on the server. To implement the restore functionality on a secondary device, the application executes a davCollection.get(), downloads the JSON file, and merges the data back into the local Room database, effectively syncing the reading progress across devices.25
To ensure synchronization occurs reliably without degrading the user experience or draining the battery, all WebDAV interactions are encapsulated within an Android CoroutineWorker using the WorkManager API. WorkManager allows the application to define execution constraints; for instance, the periodic synchronization task is configured to execute only when the device is connected to an unmetered Wi-Fi network and has sufficient battery.30 If a network request yields an HTTP 403 (Forbidden) or HTTP 500 (Internal Server Error) due to server load, the WorkManager automatically queues the task for a delayed retry using exponential backoff.32
Advanced Stretch Goal: Server-Side Generated SMIL and Media Overlays
The stretch goal of implementing server-side generated Synchronized Multimedia Integration Language (SMIL) for advanced Media Overlays represents the most sophisticated architectural challenge in this document. The EPUB 3 Media Overlays specification defines a standardized methodology for synchronizing textual content with pre-recorded audio tracks. It maps specific HTML fragment identifiers (e.g., chapter1.html#paragraph1) to precise audio timestamps (e.g., track1.mp3#t=0,15).34
Readium natively supports parsing EPUB 3 Media Overlays. During the parsing phase, the Streamer module detects SMIL files bundled within the EPUB archive. It extracts the timing data and translates the XML-based SMIL syntax into a JSON-based structure, which is then injected into the Readium Web Publication Manifest in-memory.36 When the Publication is rendered, the Media Navigator interprets this JSON manifest, streaming the audio and utilizing the Decorator API to highlight the text in perfect synchrony with the human narration.35
However, standard reading systems—including Readium—are strictly designed to parse SMIL data authored and embedded locally within the physical EPUB file. They do not automatically fill "holes" in media overlays with TTS, nor do they natively query remote servers for synchronization data.37 Supporting server-side generated SMIL requires an architectural paradigm shift: the application must dynamically inject remote audio references and synchronization timing into an EPUB that structurally lacks them.
Dynamic Manifest Mutation Architecture
To achieve this stretch goal without forking the entire Readium Kotlin Toolkit, the application must intercept the publication parsing process and execute an in-memory mutation of the Readium Web Publication Manifest.
Interception via TransformingFetcher: When the user opens an EPUB, the Readium Streamer generates the initial Publication object and its Manifest.38 The architecture implements a TransformingFetcher to intercept the pipeline before the EpubNavigatorFragment is instantiated.39
Content Extraction and Server Communication: The application utilizes Readium's extraction APIs to pull the raw text from the current chapter's HTML nodes.1 It transmits this textual payload, alongside unique identifiers, to the remote server.
Cloud-Based Synthesis: The remote server processes the text utilizing a high-fidelity cloud text-to-speech model or matches the text against an existing audiobook database. The server dynamically generates the SMIL XML document and hosts the resulting audio files, returning the SMIL data and the remote audio URIs to the application.
Programmatic Injection: Upon receiving the server response, the application parses the SMIL timing data. It programmatically mutates the in-memory Manifest of the Publication object. It appends new JSON-based media overlay definitions into the reading order, mapping the local EPUB's HTML fragment IDs to the remote audio URIs.35
Execution and Playback: Finally, the mutated Publication is passed to the navigators. Because Readium's playback and rendering architecture relies entirely on the internal JSON manifest rather than the physical files, the engine seamlessly accepts the injected data. It streams the remote audio tracks and executes the synchronized highlighting using the same mechanisms applied to local media overlays, fulfilling the complex requirement without requiring extensive modifications to the core toolkit.35
Comprehensive Implementation Strategy for LLM-Assisted Development
The highly decoupled, heavily abstracted nature of this architecture demands a rigorous, phased implementation strategy. When utilizing a Large Language Model (LLM) for code generation, submitting massive, monolithic prompts often results in the AI hallucinating non-existent dependencies, mixing legacy code with modern APIs, or suffering from architectural drift.
To mitigate these risks, the implementation plan is partitioned into granular, highly specific phases. By feeding these distinct prompts to the LLM sequentially, the developer can construct the application systematically. This ensures each architectural layer is fully functional, thoroughly tested, and accurately conforms to Readium 3.0 standards before introducing subsequent complexities.

Conclusion
The architecture delineated in this technical design document establishes a robust, highly extensible foundation for a modern Android e-reader application. By strictly adhering to unidirectional data flow and leveraging Jetpack Compose, the application achieves a highly reactive user interface, while complex interoperability patterns ensure seamless integration with Readium's imperative EpubNavigatorFragment. The sophisticated synchronization between the standalone TtsNavigator and the visual presentation via the Decorator API provides an unparalleled accessibility experience, dynamically highlighting text as it is spoken by the native Android TTS engine.
Furthermore, the integration of Room for local persistence and dav4jvm within Android's WorkManager guarantees that user data, precise reading positions, and analytical statistics are reliably and silently synchronized with a private Nextcloud instance over WebDAV. Finally, the theoretical architecture proposed for dynamic SMIL injection paves the way for advanced, server-side generated Media Overlays, ensuring the application remains at the forefront of digital reading technology and fully prepared for future expansion.
